Lesson 4: User-Defined Classes
Introduction
In this lesson you will learn how to define your own persistent-capable classes.
About schema development
The first step in developing an object-oriented application is to define the classes that capture the structure and behavior of the fundamental entities in the application. Such definitions usually arise naturally out of the logical modeling phase of application development.
In the logical model for a database application, you identify the classes whose instances are to be persistent—that is, must continue to exist even after the applications that define or manipulate them have finished. With classes whose instances will be persistent, two additional activities must be added to the class definition process:
■
Each class definition must be made persistence-capable so that instances of the class can be persistent.
■
Persistence-capable class definitions must then be added to a federated database schema so that instances of those classes can be stored in the database.
The process of defining persistence-capable classes and adding them to a schema is called schema development.
Persistence-capable classes
In a standard C++ program, instances of all classes are transient—they exist only for the life of the application that defines them.
In Objectivity/C++, you add persistence behavior to an application-defined class by deriving it directly or indirectly from the Objectivity/C++ class ooObj. Such classes are said to be persistence-capable. When a class is persistence-capable, an application can create both persistent and transient instances of it.
Classes that do not inherit persistence behavior are non-persistence-capable classes. All instances of such classes are transient. However, instances of non-persistence-capable classes are stored in the federated database when you incorporate them in a persistent object—for example, when you use a non-persistence-capable class as an attribute type or base class of a persistence-capable class. A non-persistence-capable class must obey certain restrictions to be incorporated in a persistence-capable class.
Instances of persistence-capable classes are true persistent objects, in that each can be referenced by an address in the database; this address serves as the object identifier (OID) for the object. In contrast, instances of non-persistence-capable classes are not independently addressable, although they may be accessed indirectly through the persistent objects that embed them.
Federated database schemas
Each Objectivity/DB federated database contains a schema that describes the types of data that it can store persistently. A schema contains a type number for each persistence-capable class, along with shape information, which describes how persistent instances of the class are to be laid out in storage.
Note that a schema also contains type information for any non-persistence-capable class that is directly or indirectly incorporated in a persistence-capable class. However, a schema does not contain information about member functions or static data members; such information is maintained in the compiled applications that access the database.
Objectivity/DDL
In Objectivity/C++, persistent-capable classes are defined in Data Definition Language (DDL) and saved in files. DDL is like C++ but enhanced for persistent-capable features. The DDL files can be automatically generated using Objectivity/Assist or created using an editor. Once created they can be processed by the Objectivity/DDL processor from within the Visual C++ IDE project. The DDL processor creates or updates the schema in the federated database and generates C++ source code and header files for your application.
Objectivity/Assist streamlines the creation of DDL through a graphical user interface, which is used to create the class descriptions, create the schema directly in the federated database, and optionally generate the corresponding DDL files.

Exercise 4.1A
Use Objectivity/Assist to create and add a new class RentalCompany to the default schema.
Adding a class
1.    
In the FD Project Explorer, click to expand Federated Database (RentalCompany).
2.    
Double-click Schema to open the schema overview in the editor area.
3.    
Click Add in the Default Namespace/Packages Classes section of the schema editor.
4.    
Enter RentalCompany in the Class Name field of the Add Class dialog.
5.    
Leave the following fields checked: Open editor for class when complete and Make persistence capable. Click Finish.
    
The editor area displays the schema description of the RentalCompany class. In the Base Classes section, you can see that the RentalCompany class inherits from ooObj, which is persistence-capable. This means that instances of the RentalCompany class will be stored in the database.
Adding members to a class
1.    
From the schema description of the RentalCompany class, click Add in the Members section.
2.    
From the Add Member dialog, use the Available Member Types list to select ooVString under Strings. Click Add.
3.    
Click newMember0 and replace the value with name.
6.    
Click Finish.
The RentalCompany class descrption is updated to reflect the new members.
Committing schema changes
At this point you have requested the addition of a new member, but this member is not part of the persistent schema until you commit your changes.
1.    
In the FD Project Explorer, right click Schema under the FD Project and select Commit Schema Changes.
This lists the changes you have requested.
2.    
Click Finish to commit the changes to the schema in the federated database.
Generating the DDL files
Now you can generate the DDL file for the RentalCompany class.
1.    
In the FD Project Explorer, right click Schema and select Generate Code.
2.    
For Output Directory, click Browse to navigate to installDir\samples\cxx\tutorial\RentalCompany.
3.    
Select Yes to overwrite existing files and C++ for the language.
4.    
Click Finish to generate the files.
5.    
Inspect the generated files:
■
RentalCompany\ddlFiles\RentalCompany.ddl
■
RentalCompany\src\RentalCompany.cpp
6.    
Repeat the same process to create the Vehicle and Customer classes. Reference the class diagram at the beginning of this lesson for member details.
Exercise 4.1B
Adding a derived class
Use Objectivity/Assist to create the Truck class that inherits from Vehicle in the default namespace.
1.    
In the FD Project Explorer, expand Schema, right click on Namespace (Default), and select Add Class.
2.    
Enter Truck in the Class Name field.
3.    
Clear the Make persistence capable box.
4.    
Click Finish.
5.    
In the Base Classes section of the schema description for the Truck class, click Add.
The Add Base Class dialog displays a list of all available classes from which to derive.
 
6.    
Select the Vehicle class from the list, and click Finish.
Note: The Persistence Capable field has changed from no to yes now that Truck is inheriting from a persistence capable class.
7.    
Add class members to the Truck class using the class diagram as a reference.
8.    
Commit the schema changes and repeat the process from the Generating the DDL files section to generate new DDL files.
9.    
Add ooddlx calls to your script or pre-build event to generate the required .h and .cpp files:
cd "$(ProjectDir)ddlFiles"
if exist "$(ProjectDir)ddlFiles\RentalCompany.ddl" ooddlx "$(ProjectDir)ddlFiles\RentalCompany.ddl" "$(ProjectDir)data\RentalCompany.boot"
if exist "$(ProjectDir)ddlFiles\Vehicle.ddl" ooddlx "$(ProjectDir)ddlFiles\Vehicle.ddl" "$(ProjectDir)data\RentalCompany.boot"
if exist "$(ProjectDir)ddlFiles\Truck.ddl" ooddlx "$(ProjectDir)ddlFiles\Truck.ddl" "$(ProjectDir)data\RentalCompany.boot"
if exist "$(ProjectDir)ddlFiles\Customer.ddl" ooddlx "$(ProjectDir)ddlFiles\Customer.ddl" "$(ProjectDir)data\RentalCompany.boot"
Note: The ooddlx commands are in preparation for the next lesson where you will complete your class definitions by adding associations. No application code is executed in this lesson.
