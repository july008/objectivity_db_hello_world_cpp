Lesson 7: Inspecting the Catalog
In the FD Project Explorer:
1.    
Click to expand +Federated Database to see entries for the main logical components of a federated database (Catalog, Data, and Schema).
2.    
Click +Catalog to see entries for Databases and Hosts.
The catalog makes it possible for a federated database to manage data distributed over multiple hosts in the network, and for an application to access such data.
Databases
In the FD Project Explorer:
1.    
Click to expand +Databases to see list of individual databases.
The entry for each database starts with the DB: prefix.
2.    
To see the entire entry for each database, you may need to widen the FD Project Explorer area. Move the pointer over the right edge of the FD Project Explorer. When the pointer changes shape, drag the edge to the right.
The catalog maintains a list of the databases in the federated database. Logically, databases are units of storage for your data. Physically, databases are represented as files and serve as units of distribution.
Databases are created automatically as needed by applications, under the guidance of the federated database’s placement model. The placement model encapsulates your overall design for the structure of the federated database, and determines how many databases to create, and where to create them. We’ll learn more about placement models later.
In this example, the federated database has three databases, each corresponding to a regional company of the vehicle-rental business.
You may have noticed a fourth database file listed in your directory or folder; this database contains internal housekeeping data, and so is not displayed by Assist.
3.    
Double-click on Databases to display the Databases editor in the editor area.
The table of databases displays information in columns as follows.
OID and Name columns
As logical entities, each database has an identifier and a system name. The identifier is assigned by Objectivity/DB. The system name is obtained from the federated database’s placement model. In this example, the system names indicate that each database contains the data for a particular region of the vehicle-rental business.
The identifier of a database is actually a single integer that can be expressed as a 4-component OID, or object identifier. For example, the identifier of the pmV2-Region_G1_1 database is 3, which is displayed as the OID 3-0-0-0. (The other components of OIDs are described in a later lesson.) A database’s OID and system name also appear in the corresponding DB: entry in the FD Project Explorer.
Read Only column
None of the databases in this federation is read-only; instead, these databases are read-write, permitting both read and update access to their data. Every database starts out as read-write, so you can create persistent objects in it. If these objects reach a final state, you can make the database read-only to speed up read operations. You can change whether a database is read-only at any time using Assist or an Objectivity/DB administrative command-line tool.
Page Size column
The binary contents of a database are stored on pages, which are the units of physical transfer from the database file on disk to an application process’s memory. When an application accesses data from a database, one or more whole pages are transferred to and from disk. Because the example database’s page size is 8192 bytes, its data will be transferred between memory and disk in 8192-byte chunks.
The page size for each database is obtained from the placement model, and set when the database is created.
Host and File columns
The catalog maps each logical system name onto a physical filename on a particular host computer. These properties (host and path) can be set by default, or chosen by you. You can change a database’s location at any time using an Objectivity/DB tool.
Hosts
In the FD Project Explorer:
1.    
Click to expand +Hosts to see list of individual host computers on which the federated database is distributed.
Because this federated database is a fairly simple example, all of its files and processes are located on the same host. In a more realistic deployment, the database files would be distributed among multiple hosts on the same network, so each database could be closer to the client application that accesses it most often.
That is, because each database contains data for a particular geographic region (United States, Europe, and Australia), each database could be located on a computer in the corresponding headquarters city. When databases are distributed among multiple hosts, Objectivity/Assist shows an entry for each host with a database or lock server.
2.    
In the FD Project Explorer, double-click on Hosts to display the Hosts editor in the editor area.
Notice the host of the example federated database is reported as reachable. This means that Assist can connect to the host (by "pinging" it). Of course, this host is reachable, otherwise you would not be able to run Assist on it. But when you have a widely distributed federated database, you can use Assist to help you troubleshoot unavailable data.
The lock server for the example federated database is also reported as reachable, which means it is running. Because Assist is itself an application, it must contact this lock server to obtain permission to access objects in the example federated database.
Advanced Multithreaded Server (AMS) may also be reported as reachable on your host. AMS is data-server software that is provided with Objectivity/DB. When AMS is reachable (running), it will be contacted by any applications running on other hosts that need to open, read, and fetch pages from the database files located on this host. The use of AMS is recommended, but not required. If AMS is not running on a host, applications will use other data-server software, such as the host’s operating system (for local files) or a subset of NFS (for remote files).
