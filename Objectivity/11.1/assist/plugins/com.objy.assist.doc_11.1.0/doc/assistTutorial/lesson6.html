Lesson 6: Inspecting a Federated Database
Let’s look at the properties of a federated database.
Federated-database identity
Several properties identify the federated database.
1.    
In the FD Project Explorer, click +TutorialProject to expand the entry.
2.    
Double-click Federated Database (RentalCompany) to display various properties of the federated database.
In the Properties box of the editor, notice the federated database’s system name, RentalCompany. The system name must be unique among the federated databases serviced by the same lock server.
Below the system name, the FD Number field displays the federated database’s identifier. A federated database’s identifier is a unique integer among the federated databases serviced by the same lock server.
The system name and identifier were set when the federated database was created, and they were preserved by the ooinstallfd command you entered.
You can disconnect a federated database so that it is no longer active in Assist. To do so, click the disconnect icon:
To reconnect, double-click the federated database again in the Project Explorer.
Objectivity/DB resources
Several properties represent locations of the Objectivity/DB resources that are used or managed by the federated database.
Look at the Resource Locations box in the federated-database editor.
In this list, we see the following Objectivity/DB resources.
Lock server host
The host (network node) running the lock server that services this federated database. A lock server manages concurrent access to a federated database. Applications accessing this federated database will request locks from the lock server running on this host. A single lock server can manage concurrency for multiple federated databases.
Boot file host and path
The location of the boot file. The boot file contains a list of the federated database’s properties. Applications specify the boot file to start their use of the federated database; Objectivity/DB reads the file to find the resources it needs. (Objectivity/Assist used the boot file RentalCompany.boot to open this federated database.) By convention, the name of a boot file consists of the federated database’s system name with a .boot extension.
System-database file host and path
The location of the file containing the system database. This is a special database that is managed by Objectivity/DB, and contains internal data such as the federated database’s global catalog and schema.
Journal directory host and path
The location of journal files. These files are where Objectivity/DB records uncommitted updates, enabling Objectivity/DB to recover the federated database to its last consistent state, e.g., when a transaction is aborted or terminated. Objectivity/DB creates and deletes journal files automatically.
These host and path properties were set when the federated database was created, and then adjusted by the command that installed the federated database on your host. It is possible to change any host or path property at any time, as described in the Objectivity/DB Administration online book.
